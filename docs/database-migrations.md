# Database Migrations

## Overview

We use Alembic for database migrations. Alembic is a lightweight database migration tool for SQLAlchemy, providing version control for your database schema.

## Prerequisites

Ensure you have installed the development dependencies:

```bash
pip install -e ".[dev]"
```

Or specifically:

```bash
pip install alembic
```

## Configuration

### Database URL

The database URL is configured automatically from your environment settings:

- Set via `DB_URL` environment variable
- Or configure in your `.env` file
- Default: `sqlite:///./novel_engine.db`

Supported database types:
- SQLite: `sqlite:///path/to/database.db`
- PostgreSQL: `postgresql://user:password@localhost/dbname`
- PostgreSQL + asyncpg: `postgresql+asyncpg://user:password@localhost/dbname`

### Model Registration

All SQLAlchemy models are automatically registered in `alembic/env.py`:

- `KnowledgeBaseModel`, `DocumentModel` (knowledge context)
- `StoryModel`, `ChapterModel`, `SceneModel` (narrative context)
- `WorldStateModel`, `WorldStateVersionModel`, `WorldStateSnapshotModel` (world context)

## Creating Migrations

After modifying any SQLAlchemy models, create a new migration:

```bash
# Using make
make migration message="Add user table"

# Or directly with alembic
alembic revision --autogenerate -m "Add user table"
```

This will:
1. Compare current models with the database schema
2. Generate a new migration file in `alembic/versions/`
3. Include both `upgrade()` and `downgrade()` functions

**Important:** Always review autogenerated migrations before committing!

## Applying Migrations

### Apply all pending migrations

```bash
# Using make
make migrate

# Or directly with alembic
alembic upgrade head
```

### Check current migration status

```bash
alembic current
```

### View migration history

```bash
# Using make
make migrate-history

# Or directly with alembic
alembic history --verbose
```

## Rolling Back Migrations

### Rollback one migration

```bash
# Using make
make migrate-down

# Or directly with alembic
alembic downgrade -1
```

### Rollback to a specific revision

```bash
alembic downgrade <revision_id>
```

### Rollback to the beginning

```bash
alembic downgrade base
```

## Best Practices

1. **Review autogenerated migrations**
   - Always check the generated migration file before committing
   - Ensure `upgrade()` and `downgrade()` are both correct
   - Add manual adjustments if needed for complex changes

2. **Test migrations**
   - Test migrations on a copy of production data
   - Verify both upgrade and downgrade work correctly
   - Check data integrity after migration

3. **Never modify existing migration files**
   - Once committed, treat migration files as immutable
   - If a migration is wrong, create a new one to fix it
   - This ensures consistency across environments

4. **Always provide both upgrade() and downgrade()**
   - The downgrade function allows rolling back changes
   - Test downgrade after creating the migration

5. **Keep migrations small and focused**
   - One logical change per migration
   - Easier to review and rollback if needed

6. **Use meaningful migration messages**
   - Describe what changed, not just "update database"
   - Examples: "Add user authentication tables", "Add indexes for performance"

## Migration File Structure

A typical migration file looks like this:

```python
"""Add user table

Revision ID: abc123
Revises: def456
Create Date: 2024-01-01 12:00:00

"""
from alembic import op
import sqlalchemy as sa

# revision identifiers
revision = 'abc123'
down_revision = 'def456'
branch_labels = None
depends_on = None


def upgrade() -> None:
    """Create user table."""
    op.create_table(
        'users',
        sa.Column('id', sa.Integer(), nullable=False),
        sa.Column('username', sa.String(length=80), nullable=False),
        sa.Column('email', sa.String(length=120), nullable=False),
        sa.PrimaryKeyConstraint('id'),
        sa.UniqueConstraint('email'),
        sa.UniqueConstraint('username')
    )


def downgrade() -> None:
    """Drop user table."""
    op.drop_table('users')
```

## Troubleshooting

### "Target database is not up to date"

Run migrations to update the database:
```bash
alembic upgrade head
```

### "Can't locate revision"

This usually means the database has a revision that doesn't exist in your codebase:
```bash
# Check current revision
alembic current

# If needed, stamp to a known revision
alembic stamp <revision_id>
```

### Migration conflicts in team environments

1. Pull latest changes
2. Run migrations: `alembic upgrade head`
3. If conflicts exist, coordinate with team members
4. Consider regenerating the migration after merging

## CI/CD Integration

Migrations are automatically checked in CI:

```yaml
- name: Check migrations
  run: |
    alembic current
    alembic check
```

This ensures:
- Database is in sync with migrations
- No pending migrations are forgotten

## Environment-Specific Notes

### Development
- Use SQLite for simplicity
- Migrations can be regenerated frequently
- Test both upgrade and downgrade

### Staging
- Use PostgreSQL (same as production)
- Apply migrations before deploying code
- Test with production-like data

### Production
- Always backup database before migrations
- Apply migrations during maintenance windows
- Monitor migration execution time
- Have rollback plan ready
